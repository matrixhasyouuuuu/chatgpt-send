# Проект: Живой Улей (Soft Swarm на сотах)

## 1. Суть проекта

Это не просто система запуска нескольких ИИ-агентов параллельно.

Это модель живого роя (улей/муравейник), где:
- есть одна общая цель (задача пользователя),
- есть координатор (главный агент),
- есть рабочие пчелы (child-агенты),
- есть общая среда (улей),
- есть соты (единицы знания),
- и “правда улья” рождается не из одного ответа, а из дозревания и перепроверки информации в сотах.

Ключевой сдвиг:
- мы не “заставляем рой работать” через жесткие правила,
- мы создаем среду и простые локальные правила,
- в которой полезное усиливается, сомнительное тормозится, мусор исчезает, а внимание перераспределяется.

## 2. Философия роя (обязательная, без изменений)

Секция `## Роевая задача (Swarm task)` вставляется в документацию дословно и не сокращается.

Это базовая философия проекта:
- soft swarm, а не жесткая маршрутизация,
- общая цель роя,
- overlap допустим,
- child-агенты получают общий swarm-контекст,
- координатор отвечает за запуск, сведение, сверку и итог.

## 3. Главная идея проекта (новый уровень)

Рой работает не вокруг “ответов агентов”, а вокруг `сот` (единиц знания).

Не агент является носителем истины.
Агент является рабочей пчелой, которая:
- приносит нектар (сырой факт/вывод),
- проверяет чужой нектар,
- усиливает или ослабляет соту,
- помогает дозреть знанию до “меда” (подтвержденной части общего результата).

## 4. Основные сущности

### 4.1. Координатор (главный агент)
- задает общую цель роя,
- задает стартовый контекст,
- запускает пчел,
- решает, что проверять и кем,
- отслеживает расхождения,
- принимает итоговое решение по сотам,
- собирает финальный ответ пользователю,
- при необходимости отдает общий результат специалисту (но не сырые ответы отдельных пчел).

### 4.2. Пчела (child-агент)
- это рабочий исполнитель, а не “истина”.
- одна и та же пчела может получать разные промпты/роли в разных проходах.
- пчела может быть:
- разведчиком,
- скептиком/валидатором,
- арбитром,
- уточнителем,
- сборщиком локального вывода.

### 4.3. Улей (общая среда)
- это общая память роя.
- в улье хранятся соты, их состояние, подтверждения, споры, зрелость.
- улей должен быть изменяемой средой, а не просто набором финальных отчетов.

### 4.4. Сота (единица знания)
Одна сота = один смысл.
Это важно.

Сота может быть:
- фактом,
- гипотезой,
- риском,
- наблюдением,
- выводом,
- решением.

Сота не должна содержать “весь отчет”.
Одна сота = одна проверяемая мысль.

### 4.5. Нектар и мед
- Нектар = сырой вывод/наблюдение/утверждение.
- Мед = зрелая, подтвержденная, пригодная для общего результата информация.

## 5. Что хранится в соте (минимум)

Каждая сота должна содержать минимум:
- утверждение (что именно сказано),
- доказательство (лог/код/`path:line`/наблюдение),
- источник/контекст (откуда взято),
- важность для общей цели,
- текущую уверенность,
- статус зрелости,
- историю подтверждений/оспариваний,
- отметку спорности (если есть конфликт).

Важно:
- одинаковый текст не равен одинаковой сути,
- считать нужно подтверждение одного и того же утверждения, а не совпадение формулировок.

## 6. Жизненный цикл соты (зрелость знания)

Правда улья не бинарная (“истина/ложь”), а по зрелости.

Примерный цикл:
- `сырой нектар` (первое внесение в соту)
- `дозревает` (идет перепроверка, уточнение, добавление доказательств)
- `подтверждено` (сота достаточно устойчива)
- `в общий мед` (входит в итог координатора)

Если есть конфликт:
- `спорная сота` (отдельно от подтвержденных)
- дальше сота:
- либо дозревает,
- либо дробится,
- либо уходит в арбитраж,
- либо отбрасывается.

## 7. Что делают пчелы с сотой

Пчелы работают не только на подтверждение.

Они могут:
- подтверждать (усиливать соту),
- оспаривать (ослаблять соту),
- уточнять (добавлять доказательства и детали),
- дробить (если сота слишком широкая и смешивает несколько смыслов),
- сливать (если это дубликат другой соты),
- помечать как спорную,
- возвращать на добор фактов.

Это делает рой живым.
Иначе будет просто “многократное повторение одного вывода”.

## 8. Проверка правдивости: модель с координатором (ваша ключевая идея)

Истину не определяют пчелы голосованием.
Истину собирает координатор на основе проверок и доказательств.

Базовый цикл по одной соте:
1. Пчела-1 приносит нектар в соту (утверждение + доказательство).
2. Координатор сразу запускает Пчелу-2 на независимую проверку этой же соты.
3. Если Пчела-1 и Пчела-2 сходятся по сути и доказательства сильные:
- координатор усиливает соту.
4. Если есть расхождение:
- координатор запускает Пчелу-3 (арбитражная проверка).
5. Координатор сам принимает решение:
- подтвердить,
- оставить спорной,
- вернуть на добор,
- дробить,
- отбросить.

Важно:
- счет “сколько пчел согласились” — это сигнал,
- но не единственный критерий.
- координатор учитывает:
- независимость проверок,
- качество доказательств,
- влияние на общую цель,
- наличие противоречий.

## 9. Независимость подтверждений (критично)

`3 одинаковых ответа` не означают правду автоматически.

Сильное подтверждение — это:
- независимая проверка,
- по первичным данным,
- с явным доказательством,
- без копирования чужого вывода.

Слабое подтверждение — это:
- пересказ чужого ответа,
- повтор без доказательств,
- совпадение только по формулировке.

## 10. Selection Pressure (отбор) — что усиливается, что умирает

Живой улей работает через отбор.

В рое должно быть:
- усиление полезного,
- торможение сомнительного,
- удаление мусора,
- переключение внимания.

Сота усиливается, если:
- есть независимые подтверждения,
- есть доказательства,
- сота влияет на общую цель,
- сота уменьшает неопределенность.

Сота ослабляется, если:
- нет доказательств,
- найдено противоречие,
- это дубликат,
- это шум/размытый вывод,
- долго нет подтверждения.

Сота умирает/затухает, если:
- не подтверждается в течение нескольких циклов,
- опровергнута,
- утратила релевантность текущей цели.

## 11. Координатор как “разум улья”, а не диспетчер микрошагов

Координатор не должен вручную управлять каждым движением пчелы.

Координатор должен:
- задавать общую цель,
- задавать начальный контекст,
- отслеживать состояние сот,
- направлять проверки по конфликтам и важным сотам,
- запускать арбитраж при расхождении,
- переключать внимание роя,
- собирать мед (а не сырой нектар) в итог.

Это не жесткая командная цепочка.
Это управление состоянием среды и качеством результата.

## 12. Роли пчел (через промпты, без жесткой привязки)

Одна и та же пчела может выполнять разные роли в разные моменты.

Примеры ролей:
- Пчела-разведчик:
  - ищет новые соты (новые факты/гипотезы)
- Пчела-скептик/валидатор:
  - получает чужую соту и проверяет по первичным данным
  - промпт: “не верь, проверь сам”
- Пчела-арбитр:
  - разбирает расхождение между двумя проверками
- Пчела-уточнитель:
  - не подтверждает/оспаривает, а добавляет недостающие доказательства
- Пчела-сборщик локального вывода:
  - собирает несколько зрелых сот в один кусок для координатора

Важно:
- это не фиксированные классы агентов,
- это режимы работы (промпты), которые назначает координатор.

## 13. Этажи (опционально) и связь с сотами

Этажи можно использовать, но они не являются ядром модели.

Если этажи используются, то:
- они работают над сотами,
- а не над “отдельными агентами как собственниками задач”.

Пример:
- проход 1: сбор нектаров (новые соты)
- проход 2: скептическая проверка сот
- проход 3: арбитраж/дозревание/сбор меда

Главная единица остается сота, а не этаж.

## 14. Что получает специалист (важный инвариант)

Специалист не должен получать сырой ответ одного агента как “истину”.

Специалист получает:
- общий результат от координатора,
- зрелые соты,
- спорные соты (если нужна внешняя проверка),
- контекст и цель роя.

Это сохраняет качество и не ломает общую задачу.

## 15. Минимальные локальные правила пчел (простые, живые)

Пчеле достаточно нескольких простых правил:
- работай на общую цель роя,
- считай чужой вывод потенциально ошибочным, пока сам не проверил,
- приноси в соту не мнение, а утверждение с доказательством,
- не затирай конфликт — помечай спорность,
- если сота слишком широкая — предложи дробление,
- если видишь дубликат — предложи слияние,
- если не хватает данных — явно пометь, что нужно проверить.

Этого достаточно, чтобы рой начал самоорганизовываться.

## 16. Как выглядит “правда улья”

Правда улья — это не один красивый ответ агента.

Это:
- набор зрелых сот,
- с известной историей подтверждений,
- с явными спорными местами,
- с контролируемой неопределенностью,
- собранный координатором в общий ответ.

Именно это и есть “мед”.

## 17. Почему эта модель сильнее обычного мультиагентного запуска

Обычный подход:
- много агентов,
- много отчетов,
- координатор тонет в тексте,
- проверки запаздывают.

Подход “улей + соты”:
- единица работы и проверки — сота,
- знания дозревают,
- конфликты видны отдельно,
- шум не копится бесконечно,
- координатор собирает зрелое,
- рой самоорганизуется вокруг общей среды.

## 18. MVP (без сложной автоматизации, вручную)

Первый рабочий вариант можно делать даже без сложного кода:
- координатор ведет соты вручную (таблица/список),
- пчелы получают промпты по ролям,
- каждая важная сота проходит через проверку,
- при расхождении запускается третья пчела,
- координатор решает статус соты,
- в финал идет только зрелое и явно помеченное спорное.

Это уже даст резкий рост качества без перегрузки системы.

## 19. Риски и чего избегать

Нельзя превращать улей в жесткий конвейер:
- не привязывать агентов навсегда к роли,
- не считать совпадение текста доказательством,
- не копить все подряд без затухания,
- не смешивать спорное с подтвержденным,
- не отдавать специалисту сырой единичный ответ.

Главный риск:
- 3 раза проверить одну и ту же ошибку одинаковым способом.

Противоядие:
- разные промпты проверки,
- независимые источники,
- арбитраж по расхождениям,
- координатор как финальный судья по соте.

## 20. Итоговая формула проекта

- Агенты = пчелы
- Общая среда = улей
- Единицы знания = соты
- Сырой вывод = нектар
- Перепроверка = дозревание
- Подтвержденный результат = мед
- Координатор = разум улья
- Общий ответ пользователю = мед улья, а не случайный нектар одной пчелы
