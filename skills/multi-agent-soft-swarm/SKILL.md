# Мультиагентный Soft Swarm (Specialist Handoff)

Базовое позиционирование этого skill:
- Во-первых: гибкий мультиагентный workflow координации (soft swarm) и для обычных задач разработки.
- Во-вторых: возможность handoff в браузер/Specialist для глубоких автономных исследовательских/итерационных циклов, когда это нужно.

Алиасы активации (считать немедленной активацией этого skill):
- "режим пчелы" / "режим пчёл"
- "режим роя" / "рой"
- "пчелы" / "пчёлы" (when used as workflow mode)
- "swarm" / "soft swarm"
- "multi-agent" / "мультиагенты"

## Глоссарий

- **Specialist**: чат ChatGPT Web в браузере (`chatgpt.com/c/...`).
- **Codex terminal**: только короткие сообщения пользователя + короткие статусные апдейты.
- **Bridge tool**: `chatgpt_send` (сначала искать в `PATH`, fallback — абсолютный путь к бинарнику).

## Определение Bridge-команды (обязательно)

Перед первой командой к Specialist в текущем ходе один раз определить путь к bridge binary:

```bash
if command -v chatgpt_send >/dev/null 2>&1; then
  CHATGPT_SEND_BIN="$(command -v chatgpt_send)"
elif [[ -x /home/matrix/projects/chatgpt-send/bin/chatgpt_send ]]; then
  CHATGPT_SEND_BIN="/home/matrix/projects/chatgpt-send/bin/chatgpt_send"
else
  echo "chatgpt_send not found"
fi
```

Правила:
- Если fallback-путь существует, использовать его молча и продолжать.
- Спрашивать пользователя про установку/путь только если обе проверки не прошли.
- Для всех bridge-действий в этом skill запускать `$CHATGPT_SEND_BIN ...`, а не голый `chatgpt_send`.

## Использование Skill (технически, без UX-сценария)

Этот skill — технический workflow для координатора, который умеет:
- работать в local-first режиме,
- запускать реальных child-агентов через `spawn_second_agent`,
- опционально использовать Specialist/браузер через `$CHATGPT_SEND_BIN`,
- агрегировать результаты child-агентов и продолжать работу новыми волнами.

Используйте этот skill как описание возможностей + набор инструментов исполнения, а не как жесткий скрипт общения с пользователем.

Роль агента при активации этого skill (важно):
- Ты в этом skill = координатор роя (главный агент / "мозг" роя).
- Твоя основная рабочая сила в этом режиме — рой (child-агенты / "пчёлы"), а не только одиночная работа вручную.
- По умолчанию отвечайте пользователю и принимайте решения от лица координатора, если явно не сказано иное.
- Child-агенты — это отдельные исполнители, которых координатор запускает и контролирует.
- Координатор не перестает быть координатором только потому, что умеет сам выполнять часть работы локально.
- Думай как координатор мультиагентной системы: твоя задача — быстро и надежно организовать выполнение через рой, а не просто "быть умным одиночным исполнителем".

Расширенное позиционирование роя (дополнение, не замена):
- Рой — это основной подход работы, а не режим только "для исправления кода".
- Рой — это основной инструмент выполнения задач в этом skill: через него координатор ускоряет сбор информации, проверки и параллельные действия.
- Пользователь может прийти с любой задачей: код, анализ, стратегия, идея, декомпозиция, сайт, исследование, проверка гипотез, план действий.
- Задача координатора — не мучить пользователя лишними вопросами и не перекладывать на него внутреннюю координацию роя.
- Если пользователь не дал лишних деталей, сначала действуй по имеющемуся запросу и доступному контексту, а не начинай с допроса.
- Координатор сам выбирает лучший способ работы: делать сам, запускать рой, комбинировать волны, менять число child-агентов.
- Внутреннее правило мышления: сначала думай "как это быстрее/надежнее решить роем", а потом уже решай, что делать локально самому.
- Пользователь пользуется системой как "мозгом роя": координатор сам принимает технические решения о том, как быстрее и надежнее решить задачу.
- Уточняющие вопросы задавай только когда без них реально нельзя надежно продолжать (нехватка данных, неоднозначный контекст, риск неверного действия).

## Режим мультиагентного координатора

Используйте этот режим, когда пользователь просит запустить нескольких child-агентов.

### Роевая задача (Swarm task)

Роевая задача — это режим soft swarm (мягкого роя), где несколько child-агентов работают параллельно над одной общей целью, а главный агент выступает координатором (мозгом роя), который распределяет фокус, отслеживает пересечения, сверяет результаты и принимает следующие решения.

Важно:
- Рой — это не жесткая маршрутизация и не жесткие блокировки по файлам.
- Все агенты работают на одну общую задачу, но каждый может идти своим планом в рамках своего текущего фокуса.
- Пересечение (overlap) между агентами допустимо и иногда полезно, но оно должно быть осознанным и явно отражено в отчете.

Критерий “это действительно рой”, а не просто параллельный запуск:
- Запуск нескольких child-агентов без общего swarm-контекста (`--team-goal`, `--agent-name`, `--peer`) — это только параллельное выполнение, а не настоящий режим роя.

Что обязательно получает каждый child-агент в режиме роя:
- общую цель роя (`--team-goal`)
- свою роль/фокус (что именно он сейчас исследует/делает)
- информацию о других агентах и их задачах (`--peer`, repeatable)
- правило на overlap: если обнаружено пересечение, не дублировать работу вслепую, а переключиться на проверку/валидацию/интеграцию и явно сообщить о пересечении

Что должен понимать каждый child-агент:
- кто он в этом рое (идентичность/роль)
- какую часть задачи он ведет сейчас
- кто еще участвует в связке
- что делают другие агенты
- что делать при пересечении с чужой задачей

Роль координатора:
- запускать child-агентов с полным swarm-контекстом
- следить за прогрессом и пересечениями
- сверять конфликтующие результаты
- при необходимости запускать дополнительную волну (verification/arbitration)
- делать финальную сверку и итоговый отчет пользователю

### Снимок роя (обязателен для каждого child)

Перед запуском каждого child-агента в режиме роя координатор должен передать "снимок роя" (swarm snapshot) в тексте child task и через swarm context flags.

Снимок роя должен описывать состояние роя на текущий момент:
- общая цель роя
- текущая фаза роя
- состав роя (`N` агентов, включая этого child)
- кто этот child (id/имя/роль)
- кто еще участвует и что делает каждый (`--peer`)
- правило overlap (при пересечении не дублировать вслепую, перейти в validation/integration и явно сообщить overlap)

Минимальный swarm context для запуска child в режиме роя:
- `--agent-id`
- `--agent-name`
- `--team-goal`
- `--peer` (repeatable)
- `--swarm-context-required` (launcher preflight: не запускать child без полного swarm context)

Правило именования child-агентов (задает координатор до запуска):
- `--agent-name` рекомендуется делать семантическим и отражающим роль/срез задачи (например: `Поиск-Логов`, `Watcher-Метрики`, `Server-RootCause`, `Риски-API`).
- Placeholder-имена для `--agent-name` (`agent-1`, `agent-2`, `child-1`, `child-2`) допустимы, но нежелательны: launcher выдаст warning, а не запрет.
- `--agent-id` может быть техническим; при желании координатор может делать его более читаемым (`logs-search`, `watcher-metrics`, `server-root-cause`), если это помогает координации.

Если child не получил снимок роя, такой запуск считается параллельным исполнением, а не полноценным роем.

Важная модель роя (явно):
- Это **soft swarm**, а не система с хардкодным роутингом/локами.
- Основной агент — координатор/"мозг" (можно думать как о "матке"/ядре интеллекта роевого workflow, а не о жестком диспетчере).
- Координатор не ограничен фиксированным числом агентов в тексте skill:
  - для маленькой задачи может использовать ноль или одного child-а,
  - для большой задачи может использовать несколько волн и много child-агентов,
  - число/роли выбираются динамически под задачу.
- Координатор динамически решает:
  - сколько child-агентов запускать,
  - как делить/переделять работу,
  - когда переспросить того же child-а,
  - когда добавить еще одного child-а для verification/arbitration,
  - когда продолжать разговор со Specialist, а когда отдавать работу рою,
  - когда остановиться, сделать сводку или запустить еще одну волну.
- Если результаты конфликтуют, неполные или выглядят рискованными, координатор должен сделать еще один роевой шаг:
  - переспросить того же child-а, или
  - запустить дополнительного child-а для проверки/арбитража, или
  - заново разделить задачу и запустить новую волну с обновленным контекстом.
- Child-агенты получают общий контекст (цель + peers), могут пересекаться и обязаны репортить overlap.
- Координатор отвечает за проверку результатов child-агентов, сверку расхождений и решение следующего шага.
- После того как пользователь дал **конкретную нетривиальную задачу**, поведение по умолчанию должно быть таким:
  - сделать короткий split-triage только если нужно выделить независимые срезы (папки/сервисы/группы логов/компоненты)
  - рано запускать child-агентов, когда есть независимые срезы
  - избегать вопроса "с какой области начать?", если координатор может разделить и стартовать параллельно
- Если задача естественно распараллеливается (например, несколько сервисов/групп логов/компонентов), предпочитать реальное выполнение роя (swarm execution) вместо затянутого расследования только силами координатора.
- Задачи анализа в режиме только чтения (логи, аудиты, диагностика) тоже могут быть роевыми задачами, если делятся по независимым источникам/компонентам.

Правила планирования:
- Держите один вопрос за раз.
- В автономном режиме координатора не задавайте планировочных вопросов без блокера; выбирайте и кратко озвучивайте значения по умолчанию.
- Не хардкодьте фиксированные размеры роя/лимиты в тексте skill; координатор выбирает это под задачу.
- Перед запуском кратко повторите выбранный план (и просите `confirm` только если пользователь хочет подтверждение или план заметно меняет риск).
- Когда подходят несколько skills, сохраняйте разделение ролей:
  - координатор использует этот skill для оркестрации и агрегации
  - execution-heavy skills (debugging/analysis/testing) обычно должны применять child-агенты или они используются на финальной верификации, а не как повод координатору перестать быть координатором
- План координатора с несколькими "ветками" — это только планирование; это **не** считается swarm execution, пока child-агенты реально не запущены и не отслеживаются.

Протокол запуска:
- Запускайте каждого child-а через `spawn_second_agent`.
- В режиме swarm/bee всегда передавайте `--swarm-context-required`, чтобы launcher блокировал наполовину настроенные запуски child-агентов.
- Если `spawn_second_agent` нет в `PATH`, сразу используйте абсолютный путь в репо (`/home/matrix/projects/chatgpt-send/bin/spawn_second_agent`); не считайте отсутствие в `PATH` блокером.
- "Swarm started" означает, что была предпринята попытка запуска хотя бы одного child-а, и координатор отслеживает его run-state (`PID_FILE`/`EXIT_FILE` и т.д.).
- Используйте `--launcher window`, чтобы у каждого child-а могло быть свое окно терминала.
- Всегда запускайте child-агентов в параллельном режиме (в UX нет последовательного режима запуска).
- Всегда передавайте `--skip-git-repo-check` при запуске child-агентов.
- Всегда передавайте явный `--iterations` для каждого child-запуска.
- Если режим iterations = `per-child`, передавайте значение этого child-а.
- Если режим iterations = `shared`, передавайте общее значение для всех child-агентов.
- Сопоставляйте browser policy каждого child-а с флагами:
  - `required` -> `--browser-required --open-browser`
  - `optional` -> `--browser-optional` (open-browser опционален)
  - `disabled` -> `--browser-disabled --no-open-browser`
- В режиме shared browser держите один общий Chrome profile/cookies и отдельный child state/chat metadata для каждого child-а.
- Не форсируйте отдельные окна браузера для каждого child-а; предпочитайте один видимый браузер с несколькими вкладками/чатами.
- Для политик `required` и `optional` держите `--init-specialist-chat` включенным, чтобы у каждого child-а был свой выделенный чат Specialist.
- URL выделенного child-чата должен быть сохранен в child-логе/metadata до глубокой работы child-а.
- Запускайте без `--wait`.
- Собирайте `RUN_ID`, `LOG_FILE`, `LAST_FILE`, `EXIT_FILE`, `PID_FILE` для каждого child-а.
- Сразу после запуска валидируйте старт: `PID_FILE` существует и `kill -0 <pid>` успешен. Иначе сразу помечайте child как `failed`.
- Опрос каждого child-а делайте по `EXIT_FILE` + живости процесса:
  - статус `running` допустим только когда `kill -0 <pid>` == true.
  - если процесс мертв и `EXIT_FILE` отсутствует -> статус `failed` (не репортить `running`).
  - если `running`, но `LOG_FILE` mtime не меняется >=60s -> статус `stuck`, приложить последние 20 строк лога.
- Во время опроса также забирайте финальную строку из `LAST_FILE`, когда она доступна.
- Для каждого child-а с policy `required` вставляйте browser brief прямо в текст child-задачи, чтобы child отправил этот контекст Specialist до глубокой работы.
- Отправляйте короткие строки прогресса в терминал:
  - `agent-2 running`
  - `agent-2 done: <short child result>`

Протокол агрегации:
- После завершения всех child-агентов создайте одну сводку, включающую:
  - child id
  - путь к проекту
  - browser policy
  - статус (`ok` / `failed`)
  - доказательство использования браузера (`CHILD_BROWSER_USED: ...`)
  - финальный `CHILD_RESULT`
- Всегда отправляйте эту сводку пользователю.
- Specialist отправляйте только если main browser mode = `yes`.
- Если пользователь попросил следующую итерацию, продолжайте от этой сводки.

Обработка ошибок:
- Если статус child-а показывает browser policy failure или в логах есть `chatgpt_send failed (cdp status=2)`:
  - пометить этого child-а как `failed` (не считать частичным успехом)
  - попросить пользователя вывести видимую вкладку ChatGPT чата на передний план
  - перезапустить только упавших `required` child-агентов с той же задачей и policy

## Цикл итераций

Перед каждой отправкой:
- Прочитать `done/max` через `$CHATGPT_SEND_BIN --loop-status`
- Установить `i = done + 1`, `N = max`
- Добавить в начало сообщения префикс: `Iteration i/N.`
- На последней итерации добавить:
  - "This is the last iteration. Give a final answer and a verification checklist."

После каждого ответа Specialist:
- `$CHATGPT_SEND_BIN --loop-inc`
- В терминале показать статус: сколько итераций осталось.
- Если код/файлы были успешно изменены, запустить commit helper:
  - `/home/matrix/projects/chatgpt-send/bin/commit_agent --repo "<project_path>" --push`
  - Если он печатает `нечего коммитить`, молча продолжать.

После завершения цикла:
- `$CHATGPT_SEND_BIN --loop-clear`

## Команды, которые вы запускаете внутри

Никогда не используйте `xdg-open` и не завязывайтесь на другие инструменты вроде `oracle`.

Открыть браузер:

```bash
$CHATGPT_SEND_BIN --open-browser
```

Список / переключение сессий:

```bash
$CHATGPT_SEND_BIN --list-chats
$CHATGPT_SEND_BIN --use-chat last
$CHATGPT_SEND_BIN --use-chat 2
```

Отправить Specialist и дождаться финального ответа:

```bash
$CHATGPT_SEND_BIN --prompt "TEXT"
```

Запустить второго агента Codex (по умолчанию тот же browser/profile/cookies):

```bash
/home/matrix/projects/chatgpt-send/bin/spawn_second_agent --project-path "<project_path>" --task "<task_for_child>" --iterations 3 --launcher window --wait --skip-git-repo-check --browser-required
```

Примечания:
- По умолчанию child использует общий Specialist browser profile/cookies (`CDP_PORT=9222`) с отдельными child state/chat metadata.
- Child все равно может работать в изолированном режиме, если это явно нужно: `--isolated-browser`.
- Child может открыть свое окно терминала и продолжить там.
- В режиме `--wait` используйте `CHILD_RESULT=...` как финальный возврат child-а основному агенту.
- Child prompt требует явную строку про браузер: `CHILD_BROWSER_USED: <yes|no> ; REASON: ... ; EVIDENCE: ...`.

Запуск нескольких child-агентов параллельно (пример шаблона):

```bash
# child 1
/home/matrix/projects/chatgpt-send/bin/spawn_second_agent --project-path "<path_1>" --task "<task_1>" --iterations 5 --launcher window --skip-git-repo-check --browser-required
# child 2
/home/matrix/projects/chatgpt-send/bin/spawn_second_agent --project-path "<path_2>" --task "<task_2>" --iterations 5 --launcher window --skip-git-repo-check --browser-disabled --no-open-browser
# child 3
/home/matrix/projects/chatgpt-send/bin/spawn_second_agent --project-path "<path_3>" --task "<task_3>" --iterations 5 --launcher window --skip-git-repo-check --browser-optional
```

Затем мониторьте каждого child-а по `EXIT_FILE`/`LAST_FILE` из вывода команды и агрегируйте финальный статус.

Помощник auto-commit (браузер не нужен):

```bash
/home/matrix/projects/chatgpt-send/bin/commit_agent --repo "/abs/project/path" --push
```

Опциональный trigger-режим (commit только когда конкретный файл растет):

```bash
/home/matrix/projects/chatgpt-send/bin/commit_agent --repo "/abs/project/path" --trigger-file "relative/or/absolute/file" --min-growth 10 --push
```
