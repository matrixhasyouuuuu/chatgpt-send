.# Карта зрелости мультиагентной системы «рой + координатор»
*Цель документа:* единый список направлений/терминов, которыми обычно пользуются зрелые команды (как “язык аудита”). Формат специально без жёстких инструкций: в каждом пункте — **что это**, **что часто бывает**, **что может иметь смысл проверить**, **какие артефакты обычно существуют**.

---

## Как использовать (коротко)
- Можно дать документ агенту‑координатору как **чеклист областей**: он выбирает 3–5 областей и делает внутренний обзор “что есть / чего нет / риски”.
- Если нужно — пункты легко превращаются в требования или backlog, но этот файл сам по себе **не является ТЗ**.

## 0) «Каркас» и базовые стандарты (общий язык больших компаний)

### 0.1 Well‑Architected Review (облачная рамка зрелости)
**Пиллары:** Operational Excellence, Security, Reliability, Performance Efficiency, Cost Optimization (+ часто Sustainability).  
**Ключевые слова:** `Well-Architected Review`, `pillars`, `workload`, `operational excellence`, `reliability`, `security`, `cost`.

### 0.2 Архитектура и разрезание системы
**Термины:** Clean Architecture, Hexagonal (Ports & Adapters), DDD, modular monolith vs microservices, dependency inversion, bounded contexts, anti‑corruption layer, dependency graph.  
**Артефакт, который часто встречается:** **ADR (Architecture Decision Records)** — “почему мы сделали так”.  
**Ключевые слова:** `ADR`, `ports and adapters`, `bounded context`, `modular monolith`, `dependency graph`.

### 0.3 Observability / наблюдаемость
**3 столпа:** metrics, logs, traces.  
**Термины:** correlation IDs / trace_id, distributed tracing, OpenTelemetry (OTel), AI Agent Observability.  
**Ключевые слова:** `OpenTelemetry`, `trace/log correlation`, `service.name`, `telemetry pipeline`.

### 0.4 SRE‑подход
**Термины:** SLI/SLO, error budget, alerting on SLOs, dashboards, on‑call, runbooks, postmortems.  
**Ключевые слова:** `SLI`, `SLO`, `error budget`, `runbook`, `postmortem`, `incident response`.

### 0.5 Безопасность приложений (AppSec)
**Термины:** OWASP ASVS, threat modeling, trust boundaries, attacker‑controlled input, secure by design, secrets management, least privilege, RBAC, audit logs.  
**Ключевые слова:** `OWASP ASVS`, `threat model`, `trust boundary`, `secure by design`.

### 0.6 Безопасность LLM/агентов
**Термины:** OWASP Top‑10 for LLM Apps — prompt injection, insecure output handling, model DoS, supply chain vulnerabilities.  
**Ключевые слова:** `OWASP LLM01`, `prompt injection`, `insecure output handling`, `tool output sanitization`, `agent permissions`.

### 0.7 Supply‑chain security
**Термины:** SBOM, SLSA, provenance/attestation, artifact signing, защита CI.  
**Ключевые слова:** `SBOM`, `SLSA`, `build provenance`, `artifact signing`.

### 0.8 Тестирование
**Термины:** Test Pyramid, regression, e2e критичных сценариев, contract tests, load/stress, chaos testing.  
**Ключевые слова:** `test pyramid`, `regression suite`, `contract testing`.

### 0.9 CI/CD и процесс разработки
**Термины:** trunk‑based development, CI gates, feature flags, shift‑left security (SAST/SCA/DAST), secrets scanning.  
**Ключевые слова:** `trunk-based`, `CI gates`, `SAST`, `SCA`, `DAST`, `secrets scanning`.

### 0.10 12‑Factor App
**Термины:** config in env, logs as event streams, dev/prod parity, disposability.  
**Ключевые слова:** `12-factor`, `logs event stream`, `dev/prod parity`.

### 0.11 Code Quality / поддерживаемость
**Термины:** linters/formatters, static analysis, type checking, cyclomatic complexity, maintainability index, dead code, god objects, dependency cycles, codeowners, coding standards.  
**Ключевые слова:** `cyclomatic complexity`, `dependency cycles`, `maintainability`, `static analysis`.

### 0.12 AI Governance / Risk
**Термины:** NIST AI RMF, GenAI risk profile, governance controls (безопасность/приватность/надёжность/прозрачность/ответственность).  
**Ключевые слова:** `NIST AI RMF`, `GenAI risk profile`, `governance controls`.

## 1) Словарь “здоровья проекта” (часто используют как быстрые линзы)
Ниже — популярные “короткие названия”, которыми удобно описывать наблюдения:

- **Code Quality / Code Smell** — запутанный/сложный код, трудно читать/менять.
- **Modularity / Single Responsibility** — каждый модуль/класс/агент делает одно; нет “свалки”.
- **Project Structure / Folder Architecture** — логичная структура папок/границ.
- **Agent Role Definition** — чётко описано: что агент делает/не делает, вход/выход.
- **Coordinator / Orchestrator Pattern** — роли управления и исполнения не смешаны.
- **Observability / Tracing** — можно проследить путь задачи end‑to‑end.
- **Structured Logging** — единый формат логов (часто JSON), пригодный для фильтрации.
- **Error Handling / Fault Tolerance** — система ведёт себя предсказуемо при ошибках.
- **Retry Logic / Fallback Strategy** — повторы/резервные сценарии не создают лавину.
- **Timeout Management** — нет “вечных зависаний”, есть лимиты времени.
- **Parallel Execution / Concurrency** — независимые задачи идут параллельно, без гонок.
- **Token Optimization / Cost Efficiency** — нет лишних токенов/дорогих вызовов без нужды.
- **Context Management / Memory Architecture** — контекст не раздувается, не теряется нужное.
- **Prompt Injection / Security Hardening** — внешние тексты не заставляют сделать “лишнее”.
- **Least Privilege Access** — доступы минимально необходимые.
- **Human‑in‑the‑Loop** — точки подтверждения для необратимых/опасных действий.
- **Testability / Unit Testing** — агенты/модули можно проверять отдельно.
- **Configuration Management / Externalized Config** — конфиги/промпты/ключи отделены от кода.
- **Code Duplication / DRY** — логика не размножена по проекту.
- **Performance Benchmarking / Latency Monitoring** — измеряется скорость и узкие места.

# Направления зрелости (1–20)

## 1) Цели и границы
**Что это:** общий фокус “зачем улучшаем” и что считать успехом.  
**Часто бывает:** проект работает, но непонятно, что улучшать первым.  
**Что может иметь смысл проверить:** 2–3 цели на период; “что нельзя ломать” (критичные сценарии); 3–5 метрик успеха.  
**Типовые артефакты:** `GOALS.md`, список критичных сценариев/пользовательских потоков.

## 2) Инвентаризация системы (карта местности)
**Что это:** перечень компонентов и потоков данных.  
**Часто бывает:** много связей, но “источник истины” неясен.  
**Что может иметь смысл проверить:** компоненты (координатор, воркеры, хранилища, очереди, интеграции, UI/CLI); путь “вход → план → инструменты → результат → отчёт”.  
**Типовые артефакты:** `SYSTEM_MAP.md`, `DATA_FLOW.md`, `INTEGRATIONS.md`.

## 3) Протокол роя (контракты между агентами)
**Что это:** единый формат событий/сообщений и идентификаторов.  
**Часто бывает:** дубли, хаос в логах, трудно расследовать сбои.  
**Что может иметь смысл проверить:** обязательные поля (`task_id/run_id/agent_id/attempt/budget/deadline/...`), правила ретраев/отмены/идемпотентности, формат событий.  
**Типовые артефакты:** `AGENT_PROTOCOL.md`, `EVENT_SCHEMA.md`.

## 4) Роли агентов и границы ответственности
**Что это:** кто планирует, кто исполняет, кто оценивает, кто пишет в память/хранилище.  
**Часто бывает:** координатор “делает всё”, агенты “иногда тоже управляют”.  
**Что может иметь смысл проверить:** описания ролей (что делает/не делает), вход/выход, запреты, критерии готовности результата.  
**Типовые артефакты:** `AGENTS.md` (каталог ролей), “role cards”, схемы вход/выход.

## 5) Оркестрация, параллелизм и конкуренция (Concurrency)
**Что это:** параллельные исполнения без гонок, дедлоков и дублей.  
**Часто бывает:** задачи выполняются дважды, “слоты” зависают, блокировки не чистятся.  
**Что может иметь смысл проверить:** ограничение параллельности, дедупликация задач, правила lock/lease, корректное завершение процессов.  
**Типовые артефакты:** `CONCURRENCY.md`, политика locks/leases, метрики очередей/слотов.

## 6) Observability: метрики‑логи‑трейсы
**Что это:** “видимость” работы системы end‑to‑end.  
**Часто бывает:** логи есть, но не склеиваются.  
**Что может иметь смысл проверить:** correlation IDs (`run_id/trace_id`), структура логов (JSON), базовые метрики (успех/ошибка/латентность/ретраи/стоимость).  
**Типовые артефакты:** `OBSERVABILITY.md`, дешборды/графики, политика логирования.

## 7) Надёжность (SRE): SLI/SLO, error budget, инциденты
**Что это:** измеримая надёжность и управляемая деградация.  
**Часто бывает:** “иногда падает” без ясной нормы и алертов.  
**Что может иметь смысл проверить:** SLI/SLO (success rate, p95/p99 latency, доля дублей, доля зависаний), алерты на SLO, разбор инцидентов.  
**Типовые артефакты:** `SLO.md`, `RUNBOOKS/`, `INCIDENTS/` (postmortems).

## 8) Ошибки, ретраи, фолбэки, таймауты
**Что это:** предсказуемое поведение при сбоях.  
**Часто бывает:** лавина ретраев, бесконечные ожидания, разные “стили ошибок”.  
**Что может иметь смысл проверить:** единая классификация ошибок, таймауты на все вызовы, backoff+jitter, фолбэки по типам отказов.  
**Типовые артефакты:** `ERROR_MODEL.md`, `RETRIES_TIMEOUTS.md`, таблица “ошибка → поведение”.

## 9) Безопасность проекта (секреты, доступы, аудит)
**Что это:** защита ключей/данных и принцип минимальных прав.  
**Часто бывает:** секреты в логах/конфигах, слишком широкие права у агентов.  
**Что может иметь смысл проверить:** secrets management, RBAC, audit logs, разделение окружений, политика доступа к инструментам.  
**Типовые артефакты:** `SECURITY.md`, реестр секретов, аудит‑следы.

## 10) LLM/Agent Security (prompt injection и опасные инструменты)
**Что это:** защита от “внешний текст заставил сделать лишнее”.  
**Часто бывает:** доверие к тексту из сети/файлов/инструментов, опасные tool‑вызовы без ограничений.  
**Что может иметь смысл проверить:** санитизация tool output, allowlists, policy layer, точки подтверждения (human‑in‑the‑loop) для необратимых действий.  
**Типовые артефакты:** `LLM_SECURITY.md`, матрица “tool → риск → ограничения”.

## 11) Supply‑chain security (SBOM/SLSA/CI hardening)
**Что это:** защищённые зависимости и сборка.  
**Часто бывает:** зависимости тянут уязвимости, сборки не воспроизводимы.  
**Что может иметь смысл проверить:** SBOM, контроль зависимостей, provenance/attestation, подпись артефактов, защита CI.  
**Типовые артефакты:** `SBOM.*`, `SUPPLY_CHAIN.md`, отчёты сканеров.

## 12) Тестирование (пирамида тестов + e2e критичных сценариев)
**Что это:** защита от регрессий и безопасные изменения.  
**Часто бывает:** тестов мало, рефакторинг страшен.  
**Что может иметь смысл проверить:** тест‑пирамида, минимальный набор e2e по критичным сценариям, регрессионный набор, контрактные тесты интеграций.  
**Типовые артефакты:** `TEST_PLAN.md`, `tests/unit`, `tests/integration`, `tests/e2e`.

## 13) CI/CD, качество на входе, “не ломать main”
**Что это:** автоматические проверки и стабильный процесс изменений.  
**Часто бывает:** качество зависит от “аккуратности автора”, выкатывать страшно.  
**Что может иметь смысл проверить:** trunk‑based, CI gates, линтер/тесты/типизация, секрет‑скан, базовые security‑чеки.  
**Типовые артефакты:** pipeline файлы, `RELEASE.md`, `ROLLBACK.md`.

## 14) 12‑Factor / конфигурации и окружения
**Что это:** предсказуемый запуск/остановка, единая конфигурация, dev/prod parity.  
**Часто бывает:** конфиги “размазаны” по коду, окружения сильно отличаются.  
**Что может иметь смысл проверить:** externalized config, единый способ запуска, лог‑поток как события.  
**Типовые артефакты:** `CONFIG.md`, `.env.example`, документация окружений.

## 15) Data management: схемы, миграции, ретеншн
**Что это:** жизненный цикл данных (состояние задач, результаты, логи, артефакты).  
**Часто бывает:** разрастание БД/файлов, нет правил хранения.  
**Что может иметь смысл проверить:** schema migrations, ретеншн/архивация, “single source of truth” для состояния.  
**Типовые артефакты:** `DATA_MODEL.md`, `RETENTION_POLICY.md`.

## 16) Backup / Disaster Recovery (DR)
**Что это:** возможность восстановиться после потерь.  
**Часто бывает:** бэкап “вроде есть”, но не проверяли восстановление.  
**Что может иметь смысл проверить:** RPO/RTO, регулярность, верификация восстановлением, сценарий DR‑drill.  
**Типовые артефакты:** `BACKUP_DR.md`, журналы проверки restore.

## 17) Resilience patterns: rate limiting / circuit breaker / bulkhead
**Что это:** защита от лавинных сбоев и внешних лимитов.  
**Часто бывает:** 429/5xx → лавина ретраев → всё падает.  
**Что может иметь смысл проверить:** лимиты параллельности, circuit breaker, bulkhead, backoff+jitter, правила ретраев на разные ошибки.  
**Типовые артефакты:** `RESILIENCE_PATTERNS.md`, единая политика лимитов.

## 18) Производительность и стоимость (LLM‑экономика)
**Что это:** где тратится время и деньги.  
**Часто бывает:** “внезапно стало дорого/медленно”, сложно понять почему.  
**Что может иметь смысл проверить:** бюджет на run, токены/кэш, дубли вызовов, длительность tool‑call, очереди/локи, выбор моделей по задачам.  
**Типовые артефакты:** `COST_PERF.md`, отчёты затрат, профили латентности.

## 19) Структура кода, модульность, техдолг
**Что это:** поддерживаемость, читаемость, отсутствие “бог‑файлов”.  
**Часто бывает:** файлы по 2000–3000 строк, циклические импорты, смешение домена и инфраструктуры.  
**Что может иметь смысл проверить:** границы модулей, dependency cycles, повтор кода, “god objects”, ADR для ключевых решений.  
**Типовые артефакты:** `ARCHITECTURE.md`, `ADR/`, метрики сложности.

## 20) Документация, DX, governance, privacy/compliance (когда проект растёт)
**Что это:** чтобы проект “жил” с командой/клиентами/инцидентами.  
**Часто бывает:** знания в голове, onboarding долгий, риски приватности в логах/промптах.  
**Что может иметь смысл проверить:** operator/user docs, runbooks, codeowners, процессы изменений, PII redaction, политика доступа.  
**Типовые артефакты:** `CONTRIBUTING.md`, `DEV_SETUP.md`, `USER_GUIDE.md`, `OPERATOR_GUIDE.md`, `GOVERNANCE.md`, `PRIVACY.md`.
